# 工作流与信息流 MVP

## 目标

把业主/租户服务请求、内部工单处理、巡检任务和处理日志串成可操作的信息流，让不同角色进入系统后知道“谁来接、谁来处理、谁来确认、下一步点哪里”。

## 本轮实现

- 新增后端只读聚合接口：`GET /api/v1/workflow/inbox`。
- 新增用户消息表：`biz_user_message`，支持未读/已读与源对象引用。
- 新增消息接口：
  - `GET /api/v1/workflow/messages`
  - `POST /api/v1/workflow/messages/:id/read`
  - `POST /api/v1/workflow/messages/read-all`
- 新增前端页面：`/workflow/inbox`，菜单名为“流程收件箱”。
- Header 增加待办图标入口，移动端不依赖侧边栏也能进入。
- 不改变现有工单、巡检状态机。
- 待办由真实业务数据派生：工单、工单日志、巡检任务；消息由工单流转事件生成。

## 客户服务请求闭环

1. 业主/租户在服务台提交服务请求，落为 `source_type=tenant_request` 工单。
2. 服务台/调度岗在“待派单”看到新请求。
3. 调度岗把工单派给物业、工程、信息化、安防等处理人。
4. 处理人在“我的处理”看到任务，进入详情执行接单、开始、完成、补充日志和附件。
5. 完成后进入“待客户确认”，由报修人或客服确认、评价。
6. 评价/关闭后形成闭环；超时单进入“超时提醒”。
7. 工单创建、派单、接单、完成、确认、评价、退回、超时等动作会给相关接收人生成消息。

## 巡检结果信息流

1. 巡检责任人在“巡检待办”看到自己的巡检任务。
2. 巡检异常由现有巡检/隐患/工单能力承接。
3. 工单处理日志形成信息流，相关角色可回到业务详情页处理下一步。
4. 本轮先做聚合视图；后续可增加 `biz_user_message` 支持已读、归档和定向推送。

## Runtime 设计

当前运行模式是 `read_model + message_sink`：

- 读取现有业务表生成待办和消息。
- 工单状态动作同步写入用户消息，不改变原有业务状态机。
- 写入动作仍由业务详情页的既有接口完成。
- 不绕过 RBAC、模块权限、SLA 和状态机。

后续增强：

- 角色订阅与部门路由规则。
- 工单/巡检/隐患事件发布。
- 移动端 Badge 与 WebSocket/轮询刷新。
