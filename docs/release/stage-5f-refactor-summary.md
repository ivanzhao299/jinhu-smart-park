# JinHu Smart Park 阶段五-F 重构治理总结报告

## 1. 总结目的

本文用于总结阶段五-F 的前端大页面拆分、后端 query service 拆分、验证结果、剩余风险和下一阶段路线。

本阶段不是功能开发总结，而是可维护性治理收口报告。报告目标是把阶段五-F 已完成的低风险拆分、收口复核、回归验证和后续建议统一归档，作为后续测试补强、接口快照设计和其它治理工作的依据。

## 2. 阶段五-F 定位

阶段五-F 是可维护性治理阶段，不是新功能开发阶段。

本阶段核心策略：

- 前端大页面低风险拆分。
- 后端纯查询 service 拆分。
- 每轮拆分后做收口复核。
- 每次只拆一个小批次。
- 不改变 URL。
- 不改变 API 路径。
- 不改变 DTO / entity。
- 不改变返回结构。
- 不改变权限、租户、园区和数据范围过滤语义。
- 不触碰账务、租赁主链、幂等、认证、migration / seed 等高风险链路。

阶段五-F 的主要目标不是把所有大文件拆完，而是在不扩大行为风险的前提下，把高频维护页面和高频查询 service 的主要职责边界先拆清楚。

## 6. 治理收益

阶段五-F 的主要治理收益：

- 主要大页面体量下降，展示 JSX 从页面主文件中拆出。
- 前端展示组件边界更清楚，页面主文件更接近 orchestrator。
- 后端查询职责从大 service 中拆出，主要只读查询有明确落点。
- 写入、状态流转、账务、幂等、认证等高风险链路未被扰动。
- controller 查询入口更清楚。
- 原 service 继续保留 facade，兼容其它模块或既有调用。
- 后续维护查询逻辑时，可以优先定位到 query service。
- release 前质量门禁和回归证据更清晰。
- 阶段性文档已经记录每一批拆分、收口判断、剩余风险和后续建议。

## 7. 剩余风险

当前剩余风险：

- 工单跨模块聚合 `tenant360Workorders` / `unitWorkorders` 仍在原 service。
- 房源跨模块大聚合仍在原 service。
- 房源资产统计大聚合和状态看板大聚合仍在原 service。
- query helper 复制后需避免长期分叉。
- 尚未新增系统化接口快照测试。
- 大 service 中仍存在状态流转和副作用复杂度。
- 浏览器级交互回归仍需后续补强。
- 文档数量增加后需要持续维护索引。
- 如果继续拆跨模块聚合，必须重新评估调用方、字段策略、数据范围过滤和返回结构。

## 8. 是否阶段五-F 收口

建议阶段五-F 当前收口。

判断依据：

- 前端资产房源页面和工单列表页面均完成低风险展示层拆分并阶段性收口。
- 后端房源 query service 完成第一批纯查询迁移并阶段性收口。
- 后端工单 query service 完成第一批、第二刀 2A、第二刀 2B，并完成阶段性总结。
- 各批次均保留路由、DTO、返回结构、权限、租户 / 园区过滤和状态语义。
- 高风险链路未被扰动。
- 继续沿同一线深拆的收益下降，风险开始向跨模块聚合、请求编排、状态流转和权限语义靠近。

不建议继续沿同一线直接拆工单 service。

不建议直接迁移跨模块聚合。

后续如评估跨模块聚合，应先只读设计，不直接实施。

## 9. 下一阶段建议

优先建议进入测试补强：

- 接口快照设计。
- `stats` 自动化对照。

- 浏览器交互回归清单。
- 前端页面主要操作路径人工或自动化走查记录。

### P2：文档索引整理

建议补强文档组织：

- release 文档索引。
- 治理阶段索引。
- 验证命令索引。
- PR 模板沉淀。
- 阶段五-F 文档入口和阅读顺序整理。

### P3：其它后端 service 只读设计

如继续后端治理，建议只做设计：

- 仅盘点，不直接实施。
- 不直接动账务。
- 不直接动租赁合同主链。
- 不直接动幂等。
- 不直接动认证。

### P4：跨模块聚合只读设计

如确需评估跨模块聚合，建议只做独立设计：

- `tenant360Workorders`
- `unitWorkorders`
- 房源大聚合

不建议直接实施。跨模块聚合拆分前必须先确认调用方、返回结构、权限 / 字段策略、数据范围过滤和回归方式。

## 10. 阶段结论

阶段五-F 已达到阶段性目标。

建议当前阶段收口，下一步转入测试补强 / 接口快照设计，而不是继续拆代码。

后续如继续治理，应继续坚持小批次、只读设计先行、行为不变、验证先行和高风险链路冻结的原则。
