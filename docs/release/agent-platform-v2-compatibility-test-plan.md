# Agent Platform V2 Compatibility Test Plan

Task: `AGENT-PLATFORM-V2-A2-VALIDATION-COMPAT`
Batch: `AGENT-PLATFORM-V2-20260621`
Owner: `agent-2`
Date: 2026-06-22

## 1. Goal

This plan defines how Agent Platform V2 proves backward compatibility while introducing event files and bounded parallel runner planning.

Compatibility means:

- Existing legacy JSON queue workflows continue to operate when events are absent.
- Event files can be introduced without making existing queue JSON unreadable.
- Generated read-model JSON remains compatible with current status, audit, runner, commit, integrate, reconcile, and validation scripts.
- Parallel runner planning can preview `--parallel 1|2|3|5`, while real parallel execution stays blocked until event-first completion writes are implemented.

This plan is documentation-only. It does not run Agents, merge, push, deploy, modify business code, add migrations, touch database state, change auth/CI/Docker/deploy/runtime configuration, or edit production environment files.

## 2. Compatibility Contracts

| Contract | Required behavior |
|---|---|
| Legacy fallback | If `ops/agent-orchestrator/events/` is absent, scripts continue to read `queue/task-queue.json`, `queue/task-locks.json`, and `queue/task-results.json`. |
| Bootstrap with events | If event directories exist, bootstrap task metadata may still come from legacy queue JSON until every task has a `task.created` event. |
| Event validation first | Event-backed apply mode validates every event before writing any generated JSON or report. |
| Generated JSON ownership | Shared queue JSON is generated by one orchestrator process, not concurrently by multiple Agents. |
| No-write modes | `--dry-run` and `--no-write` parse, validate, sort, reduce, and report without touching files or timestamps. |
| Deterministic materialization | Repeated materialization with unchanged inputs produces no diff. |
| Strict corruption handling | Invalid JSON, missing required fields, invalid timestamps, invalid Agents, unsupported schema versions, and unknown event types fail loudly. |
| Duplicate handling | Same event id/same hash warns and ignores duplicate; same event id/different hash is fatal. |
| Integration guardrails | HIGH-risk changes and non-bookkeeping conflicts require human review; queue bookkeeping conflicts are regenerated through reconciliation. |
| Parallel execution gate | `--parallel > 1` execution is blocked until completion/result writes are event-first. |

## 3. Phase Plan

### Phase C0: Current Legacy Baseline

Purpose: establish that the existing JSON queue path still works before event files are introduced.

Required checks:

```bash
node ops/agent-orchestrator/scripts/check-dispatch-status.mjs
node ops/agent-orchestrator/scripts/orchestratorctl.mjs agent-cycle --dry-run
node ops/agent-orchestrator/scripts/run-validation-matrix.mjs --plan
```

Expected:

- Commands exit 0.
- Queue, locks, and results JSON parse successfully.
- Dry-run output reports no Agent execution and no writes.
- No merge, push, deploy, production operation, migration, seed, cleanup, reset, or prune is performed.

### Phase C1: Event Directory Introduction

Purpose: prove adding event directories does not break legacy bootstrap behavior.

Fixture checks:

- Valid legacy queue JSON plus empty `events/tasks`, `events/locks`, `events/results`, and `events/audits`.
- Status projection includes all legacy tasks.
- Empty event directories do not delete READY, CLAIMED, DONE, FAILED, BLOCKED, or AUDITED tasks.
- No shared JSON file is rewritten in dry-run.

Expected:

- Event-aware scripts can distinguish "events absent" from "events present but empty".
- Legacy task metadata remains visible until `task.created` events exist.

### Phase C2: Event Read Model Dry-Run

Purpose: prove event files can generate the legacy compatibility read model without writing.

Required fixtures:

- Full happy path lifecycle.
- Failed task lifecycle.
- Blocked task lifecycle.
- Same-timestamp event ordering.
- Duplicate id/same hash.
- Duplicate id/different hash.
- Corrupt event set.

Expected:

- Dry-run prints planned writes for `task-queue.json`, `task-locks.json`, `task-results.json`, per-task result summaries, and audit summaries.
- Valid fixtures produce deterministic output.
- Duplicate id/same hash warns but exits 0.
- Duplicate id/different hash and corrupt event fixtures exit non-zero before writes.

### Phase C3: Event Read Model Apply

Purpose: prove generated JSON remains compatible with existing readers.

Apply checks must run only against fixture copies.

Expected:

- Materialized `queue/*.json` is valid JSON.
- `check-dispatch-status.mjs` can read generated queue files.
- `audit-all-results.mjs --dry-run` can audit generated result state.
- `run-validation-matrix.mjs --plan` can run after materialization.
- Re-running apply with unchanged events produces no diff.

### Phase C4: Script Adapter Compatibility

Purpose: prove each legacy script remains compatible as adapters move from direct JSON writes to event-first writes.

| Script | Compatibility proof |
|---|---|
| `claim-task.mjs` | Legacy fixture claim still works; event mode writes `task.claimed` and `lock.created`; stale/duplicate locks block. |
| `complete-task.mjs` | Legacy fixture completion still works; event mode writes `result.recorded`; wrong owner or invalid status fails. |
| `audit-agent-result.mjs` / `audit-all-results.mjs` | Legacy audit dry-run/apply behavior remains clear; event write mode records audit events and leaves final queue status to materializer. |
| `dispatch-ready-agents.mjs` | Dry-run remains no-write; apply can produce prompts and claim/lock events or fixture legacy writes. |
| `run-claimed-agent-prompts.mjs` | Dry-run planning works for accepted parallel values and prints no-execute guardrails. |
| `commit-agent-results.mjs` | Dry-run reports risk and boundary checks; HIGH-risk and forbidden paths block apply. |
| `integrate-agent-results.mjs` | Dry-run creates no branch; apply blocks unsafe candidates and regenerates queue bookkeeping from read model. |
| `reconcile-task-results.mjs` | Dry-run remains no-write; apply writes deterministic generated JSON only after validation. |
| `run-validation-matrix.mjs` | Plan mode prints commands; run mode stops on first failing command. |
| `orchestratorctl.mjs agent-cycle` | Dry-run composes read-only plans and never executes Agents. |

### Phase C5: Parallel Runner Compatibility

Purpose: prove V2-B planning does not break the safe serial default.

Required checks:

```bash
node ops/agent-orchestrator/scripts/run-claimed-agent-prompts.mjs --dry-run
node ops/agent-orchestrator/scripts/run-claimed-agent-prompts.mjs --dry-run --parallel 1
node ops/agent-orchestrator/scripts/run-claimed-agent-prompts.mjs --dry-run --parallel 2
node ops/agent-orchestrator/scripts/run-claimed-agent-prompts.mjs --dry-run --parallel 3
node ops/agent-orchestrator/scripts/run-claimed-agent-prompts.mjs --dry-run --parallel 5
```

Expected:

- Default parallelism is 1.
- Accepted values print planned batches and per-task log paths.
- Dry-run does not execute Codex and does not mutate Agent worktrees.

Negative checks:

```bash
node ops/agent-orchestrator/scripts/run-claimed-agent-prompts.mjs --dry-run --parallel 0
node ops/agent-orchestrator/scripts/run-claimed-agent-prompts.mjs --dry-run --parallel 4
node ops/agent-orchestrator/scripts/run-claimed-agent-prompts.mjs --dry-run --parallel all
node ops/agent-orchestrator/scripts/run-claimed-agent-prompts.mjs --apply --execute --parallel 2 --precheck-only
```

Expected:

- Invalid values exit non-zero before execution.
- `--parallel > 1` execute precheck exits non-zero with the event-sourced completion dependency message.
- No Codex Agent is executed.

### Phase C6: Integration Apply Guardrails

Purpose: prove V2 does not weaken integration safety.

Required cases:

- No candidate commits.
- LOW/MEDIUM docs-only candidate.
- HIGH-risk candidate touching forbidden or business paths.
- Queue-only bookkeeping conflict.
- Non-bookkeeping conflict.
- Validation failure after merge.

Expected:

- Dry-run never creates an integration branch or attempts merges.
- HIGH-risk candidates block before merge.
- Queue-only conflicts are resolved by keeping integration branch queue files and running reconciliation/materialization.
- Non-bookkeeping conflicts abort merge and require human review.
- Validation failure exits non-zero and prevents GO reporting.
- No push, deploy, production operation, migration, seed, cleanup, reset, or prune is performed.

## 4. Negative Test Requirements

The compatibility suite must include these failure classes:

| Class | Required assertion |
|---|---|
| HIGH-risk paths | Apply/execute/integrate paths stop and report human approval requirement. |
| Corrupt events | Invalid JSON and invalid schemas fail before writes. |
| Duplicate events | Same id/different hash is fatal; same id/same hash is idempotent with warning. |
| Stale locks | Missing task, wrong owner, non-active task, duplicate active Agent lock, and duplicate active task lock do not dispatch or execute. |
| Codex failure | Missing Codex and non-zero fake Codex exits are captured without launching additional tasks. |
| Validation failure | Audit, typecheck, or validation matrix failure stops integration and records failure clearly. |
| Dry-run mutation | Pre/post checksums prove dry-run and no-write paths do not mutate files. |

## 5. Release Gate Checklist

Before V2-A event read model is considered compatible:

- Legacy-only status and dry-run cycle pass.
- Event-backed dry-run projection passes for happy path, failed task, blocked task, and audit path.
- Corrupt event and duplicate conflicting event cases fail before writes.
- Generated queue JSON is byte-for-byte stable across repeated materialization.
- Current legacy reader commands can consume generated JSON.

Before V2-B parallel execution is enabled beyond preview:

- Result completion is event-first and does not require concurrent shared queue JSON writes.
- Runner summary can represent success, failed, skipped, and not-started states.
- Codex non-zero failure stops new launches and waits for started tasks to finish.
- Integration and validation guardrails pass after event-backed results are materialized.
- `--parallel > 1` execute block is intentionally removed only after the above checks pass.

## 6. Rollback And Recovery

Rollback remains available while legacy JSON is retained:

- Run status and dry-run cycle against legacy JSON when events are absent.
- Rebuild generated queue JSON from the last known good event set.
- Remove or quarantine corrupt event files in a fixture or human-reviewed repair branch before applying materialization.
- Run with `--parallel 1` or omit `--parallel` to keep serial planning/execution behavior.
- Do not merge, push, or deploy until validation and human review are complete.
