# REQ-20260621-001: Trial Launch Production Readiness Queue

## Background

The project has one orchestrator worktree and five agent worktrees. The orchestrator task pool foundation now supports natural-language intake, REQ / TECH specs, a machine-readable `task-queue.json`, claim locks, result records, and result auditing.

The current readiness source documents are:

- `docs/release/production-readiness-dry-run-report.md`
- `docs/release/production-readiness-matrix.md`

The dry-run conclusion is No-Go because engineering gates were not proven, auth mock-disabled behavior failed in local smoke, release-chain and rollback evidence was missing, and finance, safety, IoT, audit, file, and broader regression coverage remained unverified.

## User Goal

Move Jinhu Smart Park toward a trial-launch-ready state by turning the most important No-Go, Conditional-Go, and unverified readiness gaps into a formal queue that agents can claim automatically.

## Requirements

1. Generate a formal request document.
2. Generate a technical decomposition document.
3. Generate a machine-readable task queue.
4. Ensure each queued task can be claimed by the assigned agent through `claim-task.mjs`.
5. Generate only tasks in this turn; do not run real development tasks.
6. Limit the batch to 3-5 tasks with the highest trial-launch impact.
7. Prefer tasks for agent-2, agent-3, and agent-5.
8. Assign agent-4 only if RBAC, menu visibility, or dashboard acceptance requires a dedicated task.
9. Do not force an agent-1 task unless assets, units, tenants, or space data is directly in scope.

## In Scope

- Contract and finance readiness evidence.
- Receivable, payment, invoice, waiver, idempotency, and audit evidence.
- IoT, safety hazard, linked action execution, alert visibility, and energy smoke evidence.
- RBAC and menu visibility readiness evidence.
- Release acceptance, test matrix, release smoke, rollback evidence, launch gates, file/backup evidence, and production readiness reporting.
- Queue setup with all generated tasks in `READY` status.

## Out Of Scope

- High-risk business refactor.
- New migrations.
- Old migration edits.
- Auth, CI, Docker, deploy, SMS, or WeChat runtime changes.
- Real production deploy.
- Destructive seed, cleanup, database reset, or production data mutation.
- Merge or push.
- Actual formal agent task execution in this generation turn.

## Safety Constraints

- Do not modify `apps/api`.
- Do not modify `apps/web`.
- Do not modify `packages`.
- Do not modify `database/migrations`.
- Do not modify `database/seeds`.
- Do not modify `infra`.
- Do not modify auth, CI, Docker, deploy, SMS, or WeChat runtime configuration.
- Do not commit secrets, tokens, passwords, production connection strings, or real production accounts.
- Any future task that may need business code changes must set `requires_human_approval: true`.
- Merge, push, deploy, migration, production data operation, and production configuration changes require human confirmation.

## Acceptance Criteria

1. `ops/agent-orchestrator/specs/TECH-20260621-001.md` maps readiness gaps to 3-5 concrete tasks.
2. `ops/agent-orchestrator/queue/task-queue.json` contains only READY tasks, each with schema-required fields.
3. `ops/agent-orchestrator/queue/task-locks.json` contains no active locks after validation.
4. `ops/agent-orchestrator/queue/task-results.json` contains no completed results for this batch.
5. `ops/agent-orchestrator/parallel-task-board.md` lists the generated batch and each task owner.
6. `node ops/agent-orchestrator/scripts/claim-task.mjs agent-2` proves a task can be claimed, then the queue is restored to READY/no locks.
7. `node --check`, JSON parse checks, `pnpm typecheck`, and `git diff --check` pass.

## Non-Goals

- This request does not fix any product behavior.
- This request does not execute production readiness smoke.
- This request does not decide Go / No-Go.
- This request does not create PRs, merge branches, push commits, deploy, or modify production data.
