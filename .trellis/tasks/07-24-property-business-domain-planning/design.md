# 房产经营业务总体设计

## 1. 设计目标

在不破坏现有园区商业租赁、财务和资产能力的前提下，将系统演进为支持多经营主体、多项目、整套房源长短租切换的房产经营 SaaS。

## 2. 领域上下文

```text
SaaS 租户（经营主体）
└── 经营项目（园区 / 公寓 / 人才公寓 / 民宿项目）
    └── 物理资产（asset_*）
        └── 经营房源（biz_unit）
            ├── 统一经营模式与占用账本
            ├── 园区商业租赁
            ├── 民宿短租
            └── 住房长租
```

共享服务包括身份权限、文件、审计、财务底座、能源、工单、通知和统计。

## 3. 领域边界

| 领域 | 负责 | 不负责 |
|---|---|---|
| SaaS/组织权限 | 经营主体隔离、用户、角色、项目数据范围 | 租客/住客业务身份 |
| 物理资产 | 资产归属、园区、楼栋、楼层、物理空间 | 租约和预订 |
| 经营房源 | 可经营单元、用途、照片、设施、经营模式 | 具体订单和租约状态 |
| 统一占用 | 时间区间、排他占用、维修/运营锁定 | 价格、合同条款 |
| 商业租赁 | 企业客户、招商、商业合同 | 个人住房租约 |
| 民宿短租 | 日价、房态、预订、入住、退房、保洁 | 长期租金周期 |
| 住房长租 | 个人租客、住宅租约、交割、周期费用 | 按日预订 |
| 财务底座 | 应收、实收、核销、退款、审计 | 各业务的计价规则 |
| 采购成本 | 轻量采购单、成本归集、转收费来源 | 库存和仓储 |

## 4. 核心模型

### 4.1 资产与经营房源

- `asset_*` 继续表达物理资产结构。
- `biz_unit` 继续作为经营房源锚点。
- 后续通过显式映射、数据质量检查和唯一约束治理两者一致性。
- 不在本阶段进行高风险全表合并。

### 4.2 经营模式

推荐在经营房源层增加：

- `operating_mode`: `none | short_stay | long_rent`
- `operating_status`: `preparing | available | occupied | suspended | out_of_service`
- 模式切换日志：前后模式、原因、操作者、时间、检查结果

模式切换服务必须查询统一占用、业务状态和未结事项，不能仅修改枚举字段。

### 4.3 统一占用账本

推荐建立通用占用实体：

- 租户、项目、房源
- 来源类型：`long_lease | short_booking | short_stay | maintenance | operator_hold`
- 来源 ID
- 开始/结束时间或业务日期
- 状态：`held | active | released | completed | cancelled`
- 幂等键、版本、审计字段

统一采用 `[start, end)`。住房租约按日期范围占用；民宿按入住日到退房日占用；维修和运营锁房也进入同一冲突检查。

### 4.4 业务相对方

推荐抽象 `party`：

- `organization`：兼容企业客户。
- `person`：住房租客、同住人、民宿住客。

首期采用兼容适配策略：

- 保留 `biz_park_tenant` 和现有商业合同外键。
- 新业务以个人档案为主。
- 统一查询和报表通过 party reference 或读取模型聚合。
- 不在首期强制迁移全部企业客户。

### 4.5 财务来源

统一财务记录必须保存：

- `source_domain`
- `source_type`
- `source_id`
- `charge_item`
- 原始金额、调整金额和当前金额

民宿订单、住房租约、能源账单和采购转收费由各自领域计算，财务底座负责记账、收款、核销、退款和审计。

## 5. 关键状态流

### 经营模式切换

```text
none → short_stay
none → long_rent
short_stay → none → long_rent
long_rent → none → short_stay
```

禁止直接跨模式切换。必须先回到 `none`，完成未来占用、在住/租约、保洁维修和未结事项检查。

### 分批上线状态

```text
planned → developing → uat_pending → uat_in_progress
                                   ├→ uat_passed → limited_release → production_enabled
                                   └→ blocked
```

目标范围与当前开放状态分离存储和表达。

## 6. 兼容性

- 现有商业租赁继续使用 `biz_unit`、`biz_park_tenant` 和现有合同/应收表。
- 新增占用能力时，需要为现有有效商业合同回填或适配占用读取模型。
- 短租和住房长租均新增专属聚合根，不直接扩展商业合同为万能对象。
- 菜单白名单继续作为阶段性开关，但文档不再将其描述为最终产品边界。

## 7. 安全与合规

- 个人证件、手机号等敏感字段加密或脱敏，并受字段权限和审计控制。
- 所有高风险创建、确认、退款、模式切换和占用操作使用幂等或等价防重。
- 财务记录不物理删除，使用作废、冲销和反向记录。
- UAT 不降低生产级认证、迁移和财务安全规则。

## 8. 交付顺序

1. 校正范围和环境权威入口。
2. 建立共享经营模式、占用和个人业务相对方基础。
3. 实施民宿 MVP。
4. 实施住房出租 MVP。
5. 建立跨业态统一经营分析。

## 9. 主要风险

| 风险 | 缓解 |
|---|---|
| `asset_*` 与 `biz_*` 数据漂移 | 明确责任、映射和质量检查，不立即大迁移 |
| 长短租占用重叠 | 统一账本、数据库约束、事务和并发冲突翻译 |
| 万能合同/万能订单膨胀 | 保持商业合同、民宿订单、住房租约独立聚合 |
| 个人隐私泄露 | 加密、脱敏、字段权限、审计和导出控制 |
| 财务来源混淆 | 强制来源域/来源 ID，采购成本与租客应收分账 |
| 历史“生产”口径误导 | 新增当前声明，保留历史报告但解释环境语义 |
