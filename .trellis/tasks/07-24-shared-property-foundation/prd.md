# 共享房产底座领域蓝图

## Goal

在现有多租户、园区、物理资产和可出租房源模型上，建立可同时支撑园区商业租赁、民宿短租和住房长租的共享房产经营底座。

## Dependencies

- 依赖 `07-24-scope-uat-alignment` 提供范围和环境权威口径。
- 必须先于民宿 MVP 和住房出租 MVP 实施。

## Confirmed Product Decisions

- 每个经营公司/机构是一个 SaaS 租户，项目/小区/园区是租户下的经营项目。
- 租客、同住人和住客是业务相对方，不是系统租户。
- 同一套房源允许在长租和短租间切换，但不能同时经营。
- 首期只支持整套房源，房间和床位级暂缓。
- 实施顺序：共享底座 → 民宿 MVP → 住房出租 MVP → 跨业态分析。

## Requirements

### 主数据

- 保留 `asset_*` 物理资产层与 `biz_*` 经营房源层的既有边界。
- 明确 `biz_unit` 为当前经营房源锚点，不新增第三套孤立房源主数据。
- 建立物理资产与经营房源的显式映射和数据一致性规则。
- 为经营房源补充经营用途、当前经营模式、可经营状态、停用原因和模式切换记录。

### 经营模式

- 支持 `none`、`short_stay`、`long_rent` 三种当前经营模式。
- 长租与短租模式切换必须受权限、原因和审计控制。
- 切换前检查有效租约、未来预订、在住记录、待退房、待保洁、维修占用和未结财务事项。
- 模式切换不删除历史订单、租约、账单和占用记录。

### 统一占用

- 建立整套房源的统一时间占用账本。
- 占用来源至少支持长租租约、短租预订/入住、维修停用、运营锁房。
- 时间区间采用 `[start, end)` 语义，退房/租约结束日可作为下一占用开始日。
- 数据库层和服务层共同防止有效占用重叠。
- 取消、作废或完成后保留历史，但释放未来可用性。

### 业务相对方

- 建立可表达个人和组织的业务相对方概念。
- 兼容现有 `biz_park_tenant` 企业客户，不要求一次性重构历史商业租赁。
- 民宿住客、住房租客和同住人使用个人档案，并执行敏感字段权限、脱敏和审计。

### 共享能力

- 复用文件、附件、审计、权限、数据范围、幂等、通知、工单和能源能力。
- 财务层允许不同来源业务账单进入统一应收/收款底座，但必须保留来源类型和来源 ID。
- 共享底座只提供业务通用合同，不包含民宿订单或住房租约的专属状态机。

## Acceptance Criteria

- [ ] 蓝图明确 `asset_*`、`biz_unit` 及新增经营能力的责任边界。
- [ ] 同一整套房源不存在重叠的有效长租和短租占用。
- [ ] 经营模式切换具有前置检查、权限、原因和审计。
- [ ] 个人业务相对方与 SaaS 系统租户严格区分。
- [ ] 民宿和住房出租可以复用主数据、占用、财务、工单、能源和文件能力。
- [ ] 历史商业租赁合同和房源关联保持兼容。

## Out of Scope

- 房间和床位级经营。
- 物理资产与经营房源双表的全面合并。
- 民宿预订、住房租约的具体页面和状态机。
- 跨业态经营分析。
