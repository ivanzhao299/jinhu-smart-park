# 2026-07-06 生产上线加固与决策权限计划

## 上线目标

金湖 Smart Park 计划在 2026-07-06（周一）启动生产使用。本轮加固目标不是继续扩展新大模块，而是让现有系统满足真实生产启动的最低要求：

1. 管理员能够看到全部已上线模块。
2. 现场人员能够从移动端进入自己的作业终端。
3. 工程、物业、安全、招商、财务等角色的查看、填报、审批和决策权限边界明确。
4. 工程管理、巡检、隐患、工单、组织、用户、角色、字典等核心页面没有明显死按钮和空下拉。
5. 上线前能够用命令快速检查关键入口、迁移和权限矩阵是否齐备。

## 本轮已加固内容

### 1. 决策权限迁移

新增迁移：

`database/migrations/000166_go_live_decision_permission_hardening.sql`

该迁移只维护 `rel_role_perm`，不改业务表结构。覆盖真实职责分工中的 `JH_*` 角色，重点修复“角色存在但工程管理/决策权限没有落到生产权限表”的问题。

### 2. 上线检查脚本

新增命令：

```bash
pnpm go-live:check
```

检查范围：

1. 移动端现场作业入口是否存在。
2. 工程移动终端、工程项目、计划、日报、巡检、整改、验收入口是否存在。
3. 菜单配置是否包含关键上线页面。
4. 工程 RBAC、管理员可见性、隐患字典、上线决策权限迁移是否存在。
5. 典型死按钮/占位按钮模式是否残留。

该脚本不读取生产密码、不连接数据库，适合作为上线包的本地静态 gate。

### 3. 真实用户角色桥接

新增迁移：

`database/migrations/000167_go_live_real_user_role_bridge.sql`

生产 UAT 用户当前绑定的是既有岗位角色，例如 `PROPERTY_MANAGER`、`SAFETY_MANAGER`、`MAINTENANCE_ENGINEER`、`IOT_OPERATOR`、`FINANCE_MANAGER`、`INVEST_MANAGER`，而不是全部绑定 `JH_*` 新角色。因此本迁移按“岗位角色”补齐工程、工单、安全、文件上传等生产权限，避免按个人用户 ID 做特权白名单。

### 4. 管理员系统别名权限

新增迁移：

`database/migrations/000168_go_live_admin_system_alias_permissions.sql`

用于补齐 `system:read`、`user:read`、`dict:read` 等前端兼容权限，并授予 `SUPER_ADMIN` / `SYSTEM_ADMIN`，确保管理员在生产端可见系统管理、用户、角色、字典等上线必需模块。

### 5. P0 生产检查脚本

新增命令：

```bash
pnpm go-live:p0-check
```

检查范围：

1. 李荣杰、邵明洪、郑子勇、刘汉涛、陈国辉、宋乾昌 6 个真实 UAT 用户是否存在、启用、绑定角色。
2. 每个用户是否具备其岗位所需的关键工程/工单/安全/招商/财务权限。
3. 管理员角色是否具备工程和系统管理关键菜单权限。
4. 12 个核心业务字典是否有启用字典项。
5. `000164` 到 `000168` 上线迁移是否成功。
6. 生产端 14 个关键路由是否可访问。

该脚本连接本机生产容器和生产数据库，但不读取、不输出任何用户密码。

## 周一上线操作原则

1. 先开通管理员和业务负责人账号，不一次性放开全部普通用户。
2. 所有真实操作先从“查看、填报、提交、审核”四类动作验证。
3. 财务结算、服务器配置、生产运维类高风险动作暂不进入首日生产闭环。
4. 当天每个角色必须记录至少一条 UAT 结论：PASS / BLOCKED / NEEDS_FIX。
5. 出现权限缺口时优先补权限迁移，不临时在前端绕过权限。

## 生产 UAT 凭证闸门

生产 UAT 凭证只保存在 GitHub `production` 环境的 `PROD_UAT_CREDENTIALS_CSV` Secret 中，不进入 Git、镜像、迁移或构建产物。

1. 手工执行 `Production UAT Credentials` 工作流，并输入确认文本 `SYNC_PRODUCTION_UAT_CREDENTIALS`，才会同步六个受保护内测账号。
2. 临时凭证文件权限为 `0600`，同步结束后在 Runner 和生产机两侧删除。
3. 每次生产部署完成后，`Deploy Production` 会通过公网域名验证六个账号的登录、权限、菜单、关键 API 和页面路由；任一账号失败则发布工作流失败。
4. 密码轮换不放在数据库 migration 中，避免明文或固定哈希进入版本历史。
