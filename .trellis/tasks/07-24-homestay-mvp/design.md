# 民宿 MVP 技术设计

## 1. 聚合

- `HomestayRateCalendar`：房源基础价与日期覆盖。
- `HomestayBooking`：预订主单、住期、来源、取消规则快照和金额。
- `HomestayBookingNight`：逐夜价格快照。
- `HomestayGuest`：订单实际入住人。
- `HomestayStay`：入住、凭证交付和退房。
- `HomestayTurnoverTask`：退房后的保洁/检查读取模型，执行可复用工单能力。

## 2. 状态机

### 订单

```text
draft → confirmed → checked_in → checked_out
   └──────────────→ cancelled
confirmed → no_show
```

改期不是覆盖历史日期和金额，应生成调整记录并重新申请占用。

### 房态

房态由经营配置、统一占用、入住、保洁和维修聚合计算，不作为单一可随意修改字段：

```text
available / held / reserved / occupied /
dirty / cleaning / inspection / maintenance / out_of_service
```

## 3. 并发和幂等

- 创建/确认预订先申请临时占用，再在事务内激活。
- 使用订单号、幂等键和占用唯一约束防止重复订单和超卖。
- 改期先验证新住期，成功后原子切换占用。
- 取消释放未来占用，但保留订单、费用和审计。

## 4. 价格与取消

- 基础日价落在房源价格配置。
- 日期覆盖按房源和业务日期唯一。
- 订单逐夜保存日期、原价、调整价和最终价。
- 取消规则在订单确认时快照化。
- 退款只登记内部状态，不调用支付渠道。

## 5. 实名和凭证

- 入住人使用共享个人档案或订单关系快照。
- 证件号使用加密存储和脱敏读取。
- 钥匙/门卡发放记录独立于订单主表。
- 门锁、公安登记字段只作为未来集成状态，不产生首期外部调用。

## 6. 保洁

- 退房事务提交后发布内部事件或可靠任务，创建保洁工单。
- 房态在保洁完成前保持不可售。
- 项目配置决定完成后直接可售或进入检查。
- 异常通过工单关系连接维修和赔偿。

## 7. API/UI

- 管理后台：房态日历、价格日历、预订列表/详情、入住退房、退款和保洁看板。
- 移动端：今日到店/离店、实名登记、凭证交付、退房检查和保洁执行。
- 移动端必须提供卡片视图、触控操作、照片上传和离开页面前状态保护。

## 8. 兼容

- 财务通过来源域接入，不直接复用商业合同字段。
- 文件、审计、权限、工单和通知复用现有基础设施。
- OTA、支付、门锁和公安平台均通过未来适配器接入。
