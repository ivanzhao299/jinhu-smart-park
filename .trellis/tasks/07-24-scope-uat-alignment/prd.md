# 项目范围与 UAT 环境校正

## Goal

建立当前产品范围和环境定义的权威口径，消除历史“首发/二期/生产开放”表述对后续设计、测试和发布判断的误导。

## Confirmed Facts

- 当前项目仍在开发完善，尚未真实投入生产使用。
- 当前最高级别部署环境属于 UAT，但使用了部分 `production` 命名和生产级配置。
- 当前已经设计开发的全部功能均属于最终计划上线范围。
- 完整目标范围不删减，但允许模块分批 UAT、分批正式开放。
- 菜单白名单、模块授权和角色权限只表达阶段性开放状态，不定义永久产品边界。

## Requirements

- 定义本地开发、集成测试、UAT 和未来正式生产环境。
- 将当前部署描述为“production-like UAT”或“UAT”，不得声称已真实生产投运。
- 明确 UAT 可以验证生产配置和发布流程，但其运行记录不等同于真实生产运营证据。
- 为模块维护目标范围、开发状态、UAT 状态、开放状态和阻塞原因。
- 保留现有生产安全规则，包括 migration、seed、认证 mock、财务审计、幂等、备份和发布清理要求。
- 历史 release/gate 报告保留原始结论，并通过当前口径声明解释其历史语义，不批量篡改历史证据。
- 更新 README、文档索引、路线图、部署说明和交接入口中的当前口径。

## Environment Definitions

| 环境 | 用途 | 数据要求 | 发布要求 |
|---|---|---|---|
| Local | 单人开发和快速验证 | 本地 dev seed，可随时重建 | 不代表验收 |
| Integration/Test | 自动测试、迁移和跨模块回归 | 隔离测试数据 | CI/回归门禁 |
| UAT | 生产相似配置下的业务验收 | 受控 UAT 账号和可清理数据 | 生产级安全配置，不代表真实投产 |
| Production | 未来真实业务运行 | 正式业务数据 | 独立 Go/No-Go、备份、回滚和监控 |

## Module Lifecycle

- `planned`：属于目标范围，尚未完成。
- `developing`：正在开发完善。
- `uat_pending`：待进入 UAT。
- `uat_in_progress`：正在验收。
- `uat_passed`：已通过 UAT。
- `limited_release`：限制角色或项目开放。
- `blocked`：存在阻断问题。
- `production_enabled`：未来已在真实生产正式开放。

## Acceptance Criteria

- [x] 权威入口明确当前最高环境为 UAT，真实生产尚未启用。
- [x] 权威入口明确全部已开发模块属于计划上线范围。
- [x] 明确支持分批 UAT、分批开放，但不能把未开放模块移出目标范围。
- [x] 当前文档不再把菜单白名单描述为永久产品边界。
- [x] 历史报告保持可追溯，并标注其历史环境语义。
- [x] 环境文档不包含真实密码、Token 或私密连接串。

## Out of Scope

- 不重命名现有 Compose、GitHub Environment 或环境变量。
- 不执行部署、迁移、seed 或 UAT 数据变更。
- 不修改菜单和权限代码。
