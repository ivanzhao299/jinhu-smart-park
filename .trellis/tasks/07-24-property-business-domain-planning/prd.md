# 房产经营业务领域规划

## Goal

校正项目当前产品范围和环境定义，建立“共享房产底座—住房长租—民宿短租”的领域蓝图，并分别形成住房出租 MVP 与民宿 MVP 的可评审产品规划，为后续设计、开发、测试和交付提供统一依据。

## Confirmed Facts

- 项目仍处于持续开发和完善阶段。
- 当前最高级别部署环境实际属于 UAT 测试环境，尚未真实投入生产使用。
- 当前已经设计开发的功能均属于计划上线范围；历史“首发白名单”“二期暂不开放”等表述仅代表阶段性验收或开放状态，不是最终产品边界。
- 完整目标范围不删减，但允许各模块分批完成 UAT、分批正式开放；首次真实投产只开放已通过 UAT 且具备运维保障的模块。
- 后续优先基于现有园区、楼栋、楼层、房源、租赁、财务、工单、文件、权限等能力扩展住房出租管理和民宿业务管理。
- 规划实施顺序为：共享房产底座 → 民宿 MVP → 住房出租 MVP → 跨业态分析。
- 住房出租和民宿按多租户 SaaS 模式规划：每个房产运营公司、房东机构或民宿经营主体是一个系统租户，项目/小区/园区是租户下的经营项目。
- 个人租客、同住人和民宿住客属于业务相对方，不创建为系统租户；同一经营主体可以同时经营园区租赁、住房长租和民宿短租。
- 同一套房源允许在住房长租和民宿短租之间切换，但不允许两种方式同时经营；任一时点只能有一种有效经营模式。
- 首期最小可经营单元为整套房源；房间级和床位级经营暂缓。
- 住房出租 MVP 首期聚焦集中式公寓和园区人才公寓；分散式房东经营暂缓。
- 民宿 MVP 首期支持自营直订和人工录入订单；预留渠道名称、渠道订单号、同步状态等 OTA 扩展字段，但不对接 OTA 实时库存、价格和订单同步。
- 民宿 MVP 首期价格采用基础日价、日期覆盖和订单每日价格快照；复杂营销价格暂缓。
- 民宿 MVP 首期采用简单取消规则、改期差价和人工退款确认；复杂渠道取消规则暂缓。
- 民宿 MVP 纳入退房后的保洁周转闭环；复杂排班优化暂缓。
- 长租和民宿 MVP 首期只支持人工收款登记、核销和退款记录，暂不接入微信、支付宝等在线支付网关；每个 SaaS 租户是独立收款主体。
- 民宿 MVP 首期人工管理钥匙、门卡和入住凭证，暂不接入智能门锁。
- 民宿 MVP 首期完成内部实名入住登记，暂不接入公安住宿登记系统。
- 住房出租和民宿 MVP 首期只建设运营后台与现场移动端；公开找房/预订页面、租客 App/小程序、住客自助入住和其他消费者自助端暂缓。
- 住房租约 MVP 首期采用线下签署或外部签署后上传签署件，暂不接入电子签约平台。
- 住房出租 MVP 首期纳入租金、押金、水费、电费、燃气费、物业费等周期费用，以及耗材采购费用管理。
- 住房租约首期支持月付、季付、半年付、年付和自定义若干月一期，复杂阶梯租金暂缓。
- 耗材同时支持租客代购/赔偿费用和运营主体内部采购成本，两类分别记账，并允许通过受控操作将采购成本转为租客应收。
- 耗材采购首期只做采购单和成本归集，不建设库存、入库、领用和盘点系统。
- 本任务只完成产品范围、环境定义、领域蓝图与 MVP 规划，不实现业务代码。

## Requirements

### 1. 项目范围与环境定义

- 明确“已开发、UAT 验收、计划上线、正式投产”四类状态。
- 明确当前环境为 UAT，未来正式生产环境尚待建设和验收。
- 校正会影响后续判断的历史范围表述，避免将菜单白名单等同于最终产品范围。
- 为模块记录未验收、验收中、通过、限制开放和阻塞等阶段状态；菜单、权限和模块开关只用于执行阶段性开放策略。
- 保留现有生产安全规则作为未来正式投产基线，不因当前为 UAT 而降低迁移、认证、财务、数据和发布安全要求。

### 2. 共享房产底座

- 识别园区、项目、楼栋、楼层、房源、房间、床位、设施设备、空间状态等共享主数据。
- 明确共享数据的所有权、业务身份、状态边界和跨业态复用方式。
- 明确园区招商租赁、住房长租和民宿短租之间可复用与必须隔离的模型。
- 复用现有 `tenantId/parkId` 作为经营主体与项目级隔离基础，并为个人租客、同住人和住客建立业务档案边界。
- 设计受控经营模式切换；切换前必须验证不存在有效租约、未来预订、未完成入住/退房或阻断性占用。
- 共享底座可保留未来房间/床位层级的扩展位置，但住房出租和民宿 MVP 的租约、库存、价格、预订与入住均以整套房源为粒度。

### 3. 住房出租 MVP

- 定义目标用户、核心场景、业务闭环、范围、非目标和验收标准。
- 覆盖房源发布、租客、看房/申请、租约、入住交割、账单、押金、退租、维修等核心能力的取舍。
- 明确与现有 leasing、assets、energy、work-orders、files、auth/RBAC 模块的关系。
- 首期不包含分散式房东入驻、产权委托合同、房东结算、收益分成、平台佣金和多房东对账。
- 住房租约形成独立业务账单，并复用现有财务底座的应收、实收、核销、退款与审计能力。
- 租约保存计费周期、首期账单日、每期应收日、租金金额和尾期处理方式，并据此生成周期应收。
- 首期不支持复杂阶梯租金、营业额提成租金和多阶段自动涨价。
- 运营人员通过管理后台处理房源、租客、租约、账单和退租；现场人员通过移动端完成交割、收款登记、巡检和维修协同。
- 租约只有在审批通过并登记签署完成后才能生效；记录合同编号、签署状态、签署日期和附件。
- 预留电子签约平台、外部合同 ID、签署任务状态和回调字段，首期不实现实名签署、电子印章和签署回调。
- 水电费用优先关联现有能源表计和抄表读数，同时允许人工录入用量、单价或金额；账单保留计费周期、起止读数、用量和调整记录。
- 首期不接外部水电公司，不做自动代扣；费用调整、冲销和退款继续遵守现有财务审计规则。
- 租客代购、损坏赔偿、钥匙补办等形成租客应收；保洁用品、维修耗材、客用品等形成内部采购/成本记录。
- 采购成本转收费必须生成独立租客应收并保留采购来源、分摊金额、操作人和审计记录，不能直接将采购单视为租客账单。
- 采购单包含供应商、采购日期、项目/房源归属、费用分类、明细品名、规格、数量、单价、金额、票据附件、审批状态和付款状态。
- 首期不包含仓库、入库批次、库存余额、领用、调拨、盘点和最低库存预警。

### 4. 民宿 MVP

- 定义目标用户、核心场景、业务闭环、范围、非目标和验收标准。
- 覆盖房型、房间、房态日历、价格、预订、锁房、入住、退房、支付退款、保洁和维修占房等核心能力的取舍。
- 明确与长租库存、现有财务、工单、文件、认证和未来渠道集成的关系。
- 首期不实现 OTA 授权、房型映射、渠道价格、双向库存同步、渠道佣金和回调处理。
- 民宿订单形成独立业务账单；支付记录预留支付方式、渠道流水号和未来支付网关字段。
- 首期不实现商户号配置、支付回调、签名验签、自动退款和分账。
- 入住交付记录应包含凭证类型、发放时间、领取人、回收状态和操作审计，不保存明文长期门锁密码。
- 预留门锁设备 ID、临时密码任务状态和供应商扩展字段，未来通过 IoT/设备适配层接入。
- 支持一笔订单登记多位实际入住人；记录姓名、证件类型、证件号码、手机号、实名核验状态、登记时间和操作人。
- 入住人证件信息必须加密或脱敏展示，并受独立权限和审计控制；预留公安登记状态、回执号和失败原因。
- 可提供受控导出或人工登记辅助，但首期不自动向公安系统报送。
- 民宿前台/管家通过管理后台或现场移动端完成预订、入住、退房、收款登记、凭证交付和保洁协同。
- API、身份和数据权限为未来自助端预留，但不纳入首期验收。
- 支持周末、节假日和临时日期调价；订单保存逐日价格明细与总价快照，改期重新计算差价并保留调整记录。
- 首期不支持会员价、优惠券、套餐、连住折扣、渠道专享价和复杂收益管理规则。
- 每套房源可配置免费取消截止时间，以及超时后的固定金额或订单比例取消费；订单保存取消规则快照。
- 改期按新旧每日价格计算差额；退款由运营人员确认和登记，特殊减免必须记录权限、原因和审计。
- 首期不支持多层级渠道规则、不可取消价、分段阶梯取消和自动退款。
- 住客退房后房源自动进入待保洁并创建保洁任务，可指定负责人和计划完成时间。
- 保洁人员通过移动端执行、拍照、记录耗材和上报异常；完成后按项目配置进入待检查或恢复可售。
- 保洁发现损坏时可转维修工单或生成赔偿待处理项。
- 首期不支持复杂排班优化、路线规划、计件工资和机器人自动派单。

## Constraints

- 不将民宿预订订单直接等同于长期租赁合同。
- 不使用单一静态“空置/出租”状态替代短租的按日库存。
- 不在仓库中记录真实密钥、生产密码或个人敏感信息。
- 优先复用现有领域能力，但不能为复用而破坏领域边界和审计要求。
- 规划应基于当前代码、迁移、测试和正式文档，而不是仅依据历史路线图描述。

## Acceptance Criteria

- [ ] 项目范围文档明确全部已开发功能均为计划上线范围。
- [ ] 环境定义明确当前最高环境为 UAT，正式生产尚未启用。
- [ ] 领域蓝图说明共享房产底座、园区租赁、住房长租和民宿短租的边界。
- [ ] 住房出租 MVP 具有用户、流程、功能范围、数据边界、验收条件和后续阶段。
- [ ] 民宿 MVP 具有用户、流程、功能范围、库存模型、验收条件和后续阶段。
- [ ] 规划明确复用现有模块的位置，以及必须新增或隔离的领域能力。
- [ ] 复杂任务的 `design.md` 和 `implement.md` 已建立并可供用户评审。
- [ ] 尚未确定的产品决策被明确列为开放问题，不以技术假设冒充业务结论。

## Out of Scope

- 本轮不开发住房出租或民宿业务代码。
- 本轮不执行数据库迁移、seed、UAT 部署或正式生产部署。
- 本轮不接入 OTA、门锁、公安登记、支付渠道等外部系统。
- 本轮不重构现有资产、租赁或财务模块。

## Open Questions

- 无阻塞性产品问题；细节字段、状态码、API 与迁移拆分在各实施子任务中继续设计。
