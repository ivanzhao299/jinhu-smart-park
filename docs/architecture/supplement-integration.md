# 补充说明整合原则

## 1. 项目定位

本项目定位为 **产业园数字运营 SaaS 平台**，面向产业园区运营方、资产运营团队、招商团队、物业服务团队、安全管理团队和集团管理层。

金湖科创产业园是首个样板园区和默认演示园区，不是唯一客户，也不能作为后续业务逻辑、编码规则、权限规则或菜单规则的硬编码前提。

## 2. 当前系统已完成基线

当前项目已具备以下阶段性基线：

- S0 工程底座：monorepo、前后端工程、TypeScript、ESLint、基础脚本、Docker/PostgreSQL 环境。
- S1 系统基础能力：认证、用户、组织、角色、权限、字典、附件、审计、动态菜单、按钮权限、RBAC/SaaS 预留。
- S1.5 工程治理：Git 基线、seed 拆分、最小测试、e2e 烟测、迁移和 seed 执行脚本。
- S2-A 资产主数据：园区、楼栋、楼层、房源基础 CRUD 已存在或进入可用状态。

## 3. 补充说明纳入后的开发原则

后续开发必须按 SaaS 平台能力建设，而不是单项目定制系统建设。

核心原则：

- 多租户：所有核心数据必须按 `tenant_id` 隔离。
- 多园区：同一租户可管理多个园区，核心业务继续保留 `park_id`。
- SaaS 隔离 ID：`tenant_id`、`park_id` 是平台隔离口径下的字符串业务 ID，默认金湖租户为 `10000001`，默认园区为 `20000001`；实体主键仍使用 UUID。
- RBAC 开放化：角色、角色树、权限树、权限绑定不能写死在代码中。
- 模块授权：菜单、页面、接口和功能模块需要预留租户模块授权控制。
- 统一编码：新增核心实体必须具备业务 `code`，编码规则由统一编码服务维护。
- CAD/SVG 空间数字化：后续空间图、楼层图、房源图优先预留 CAD/SVG 数字化表达，不在当前阶段硬做复杂二维或三维。
- IoT：设备、点位、遥测、告警和联动作为独立阶段建设。
- 能耗：水、电、气、冷热等能耗采集、统计和账单作为独立阶段建设。
- 机器人：机器人巡检、任务、轨迹和告警作为独立阶段建设。
- BIM：BIM 模型、构件、楼层和空间关联作为独立阶段建设。
- 视频安防：视频点位、录像、AI 事件和安防联动作为独立阶段建设。
- AI 运维助手：知识库、工单辅助、告警研判和运营问答作为独立阶段建设。

## 4. 与当前已开发模块的冲突处理

为保护已完成工程资产，补充说明与当前实现存在差异时按兼容策略处理：

- 当前 `/system` 路由继续保留；补充说明中的 `/admin` 路由后续可作为别名、重定向或新导航分组兼容，不强制替换。
- 当前 `/assets/units` 路由继续保留；补充说明中的 `/assets/rooms` 后续可作为别名或行业语义扩展兼容。
- 当前统一响应格式继续保留：`{ code, message, data, request_id, server_time }`，不强制整体改为 `code=200` 风格。
- 当前 `biz_unit` 表继续保留，不强制改名为 `room`。
- 新增 `code` 字段和统一编码规则时，应兼容已有 `unit_code`、`building_code`、`floor_code` 等字段，不推倒重建，也不强制迁移为单一 `local_code`。
- 历史 S1 表中曾使用 UUID 类型的 `tenant_id`、`park_id`；S1-RBAC-STD-FIX 通过收口迁移统一为 `varchar(64)`，seed、JWT payload、`/users/me` 和 e2e 均使用 `10000001/20000001`。

## 5. 后续开发队列

建议后续按以下队列推进：

- S1-RBAC-STD：SaaS 权限开放化、统一编码、模块授权基线。
- S2-B：资产与房源增强，包括状态流转、导入导出、统计和轻量看板。
- S2-C：CAD/SVG 空间数字化、楼层平面图、房源图形化表达。
- S3 Leasing：招商、租赁、客户、线索、报价、合同前置流程。
- S4 WorkOrder：物业工单、报修、投诉、派单、SLA 和评价。
- S5 IoT：设备接入、点位、遥测、告警、联动。
- S6 Energy：能耗采集、统计、分摊、账单和能耗分析。
- S7 Robot：机器人巡检、任务、轨迹、异常上报。
- S8 Video：视频安防、视频点位、录像、AI 事件。
- S9 BIM：BIM 模型、空间构件、楼宇资产关联。
- S10 AI：AI 运维助手、知识库、运营问答、告警研判。

## 6. 后续 Codex 任务约束

后续所有 Codex 开发任务必须遵守：

- 前端使用 Next.js App Router，禁止 Pages Router。
- 前端样式使用 CSS Variables，禁止 Tailwind，禁止新增大型 UI 库。
- 图标使用 `lucide-react`。
- 后端使用 NestJS Module 化结构。
- ORM 使用 TypeORM。
- 所有受保护接口使用 JWT Guard，公开接口必须显式 `@Public()`。
- 所有删除必须软删除。
- 所有核心查询必须默认包含 `tenant_id` 隔离；业务园区数据还必须包含 `park_id` 隔离。
- 新增功能必须预留 module authorization，不把模块可用性写死在前端。
- 禁止项目特定硬编码，不允许把金湖、`JH`、固定楼栋编码、固定房源编码或固定客户逻辑写入业务判断。
