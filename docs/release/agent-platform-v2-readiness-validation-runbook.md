# Agent Platform V2 Readiness Validation Runbook

Task: `AGENT-PLATFORM-V2-A2-VALIDATION-RUNBOOK`
Batch: `AGENT-PLATFORM-V2-RESOURCE-READINESS-20260622`
Owner: `agent-2`
Date: 2026-06-22

## 1. Goal

This release runbook summarizes compatibility expectations for READY, CLAIMED, and AUDITED queue states while Agent Platform V2 transitions from shared JSON writes toward append-only events plus generated compatibility read models.

The release gate remains conservative: existing JSON queue workflows must keep working, event/read-model drift must be visible before apply, and baseline validation must keep doctor, audit dry-run, whitespace, and typecheck checks in the sequence.

## 2. Compatibility Contract

| Area | Release expectation |
|---|---|
| Legacy fallback | If `ops/agent-orchestrator/events/` is absent, queue readers continue to use `ops/agent-orchestrator/queue/task-queue.json`, `task-locks.json`, and `task-results.json`. |
| Event bootstrap | If event directories exist but are empty or partial, legacy task metadata remains visible until equivalent `task.created` events exist. |
| Read-model ownership | Shared queue, lock, and result JSON files are compatibility read models generated by one orchestrator process. Agents must not rely on concurrent shared JSON writes for parallel completion. |
| Validation first | Event-backed apply mode validates every event before writing generated JSON. Corrupt events, invalid schema versions, invalid Agents, missing required fields, or duplicate conflicting event IDs stop the write. |
| Determinism | Rebuilding read models with unchanged inputs must produce stable JSON with no diff except intentional materialization from new events. |
| Dry-run discipline | `--dry-run` and `--no-write` modes parse, validate, reduce, and report without touching files or timestamps. |
| Audit ownership | `audit-all-results.mjs --dry-run` is the release gate. `audit-all-results.mjs --apply` is the explicit state transition that records audit events and materializes AUDITED status. |
| Type safety | `pnpm typecheck` remains mandatory for readiness. Targeted E2E selection may narrow smoke coverage later, but it must not remove typecheck. |

## 3. Queue State Expectations

| State | Required compatibility evidence | No-Go conditions |
|---|---|---|
| READY | Task is visible in the queue read model with owner, priority, allowed paths, forbidden paths, and acceptance metadata. No active lock or completed result is attached. Dispatch can claim the task for its owner. | READY task missing from read model, duplicate task ID, active lock already present, unreadable task metadata, or READY task derived from corrupt events. |
| CLAIMED | Queue status is CLAIMED, exactly one active task lock exists, lock owner matches task owner, prompt/run context points to the claimed task, and no completed result has superseded the claim. | CLAIMED task without lock, duplicate active task locks, one Agent holding multiple active task locks without policy approval, lock owner mismatch, DONE/FAILED result still projected as CLAIMED, or stale lock for a non-active task. |
| AUDITED | A DONE result exists, audit passed for changed files and task boundaries, audit state is represented by audit event or legacy audit evidence, queue status is AUDITED, and no active lock keeps the task claimable. | Missing result evidence, failed audit, forbidden changed file, missing audit event/evidence, active lock still present, AUDITED status without source result, or queue state generated from corrupt events. |

READY and CLAIMED are dispatch-readiness states. AUDITED is an integration-readiness state, not a permission to merge or push by itself.

## 4. Event And Read-Model Consistency Checklist

Before declaring a V2 readiness batch compatible:

- `node ops/agent-orchestrator/scripts/orchestratorctl.mjs doctor` reports no queue/event blocker.
- `node ops/agent-orchestrator/scripts/rebuild-queue-read-model.mjs --dry-run` reports no unexpected drift.
- `node ops/agent-orchestrator/scripts/check-dispatch-status.mjs` can read the current compatibility queue and print status counts.
- `node ops/agent-orchestrator/scripts/audit-all-results.mjs --dry-run` can audit DONE results without writes.
- `git diff --check` passes.
- `pnpm typecheck` passes in an environment with dependencies available.

If any event-backed command reports corrupt events, duplicate conflicting event IDs, unsupported schema versions, invalid timestamps, invalid Agent names, duplicate active locks, or missing result evidence, stop release readiness until the event store or compatibility read model is repaired by an explicit orchestrator task.

## 5. Release Validation Order

Use this order for release-readiness validation:

```bash
git status --short
node ops/agent-orchestrator/scripts/orchestratorctl.mjs doctor
node ops/agent-orchestrator/scripts/rebuild-queue-read-model.mjs --dry-run
node ops/agent-orchestrator/scripts/check-dispatch-status.mjs
node ops/agent-orchestrator/scripts/audit-all-results.mjs --dry-run
git diff --check
pnpm typecheck
git status --short
```

Rationale:

- `git status --short` establishes the dirty-file boundary before diagnostics.
- Doctor gives broad context before narrow queue repair checks.
- Read-model dry-run checks whether event state and compatibility JSON have drifted before status or audit results are trusted.
- Dispatch status proves READY and CLAIMED compatibility.
- Audit dry-run proves DONE results can become AUDITED without writing during validation.
- `git diff --check` catches local patch hygiene before the slower typecheck gate.
- `pnpm typecheck` remains the final workspace type gate.
- The final status snapshot proves validation did not introduce unplanned changes.

## 6. Readiness Outcomes

| Outcome | Meaning |
|---|---|
| GO | READY/CLAIMED/AUDITED projections are consistent, event/read-model dry-run has no unexpected drift, audit dry-run passes, `git diff --check` passes, and `pnpm typecheck` passes. |
| CONDITIONAL_GO | Only explained non-blocking conditions remain, such as unrelated in-progress Agent worktrees or expected uncommitted task files. The owner and next action must be recorded. |
| NO_GO | Any queue/event corruption, duplicate active lock, missing result, failed audit, forbidden path change, whitespace failure, typecheck failure, or missing required evidence remains. |

## 7. Apply And Repair Boundaries

These commands can change orchestrator state and require explicit intent:

```bash
node ops/agent-orchestrator/scripts/rebuild-queue-read-model.mjs --apply
node ops/agent-orchestrator/scripts/audit-all-results.mjs --apply
node ops/agent-orchestrator/scripts/orchestratorctl.mjs doctor --fix-apply
```

Apply/repair commands must record:

- Operator and reason.
- Pre-apply doctor or dry-run output.
- Files expected to change.
- Post-apply `check-dispatch-status.mjs` and `git diff --check` result.
- Confirmation that no merge, push, deploy, production migration, production seed, database reset, Docker cleanup, or destructive cleanup was performed.

Do not use apply mode to mask an Agent task boundary failure. Boundary or audit failures should become follow-up work or human review items.

## 8. Release Record Template

Each readiness record should include:

| Field | Required value |
|---|---|
| Task or batch | Task ID, batch ID, and responsible Agent or orchestrator owner. |
| Queue summary | READY, CLAIMED, DONE, FAILED, BLOCKED, and AUDITED counts from `check-dispatch-status.mjs` or doctor JSON. |
| Event/read-model status | `rebuild-queue-read-model.mjs --dry-run` result and drift summary. |
| Audit status | `audit-all-results.mjs --dry-run` pass/fail summary. |
| Workspace hygiene | `git diff --check` result and final `git status --short`. |
| Type safety | `pnpm typecheck` result or blocked reason. |
| Exceptions | Any CONDITIONAL_GO item with owner and deadline. |
| Prohibited operations | Explicit confirmation that merge, push, deploy, production operations, migrations, seeds, destructive cleanup, and database reset were not run unless separately approved and recorded. |
